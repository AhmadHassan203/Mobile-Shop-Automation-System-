## 1. Purpose and status rules

This document is the traceability bridge between the approved static prototype and the production application. It does not promote prototype JavaScript, mock records, `localStorage`, in-memory mutations, fabricated identifiers, or toast-only actions into production behavior.

Status terms are deliberately strict:

| Status                                | Meaning                                                                                                          |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **Verified**                          | The implementation exists and the stated check was executed successfully.                                        |
| **Implemented; verification pending** | Files exist, but the complete package or cross-package quality gate has not yet run.                             |
| **Schema foundation only**            | Supporting database structures exist, but the API, authorization, audit behavior, UI, and workflow tests do not. |
| **Planned**                           | No production implementation exists. Prototype behavior remains reference material only.                         |
| **Deferred**                          | Approved for a later slice or feature flag after the core is stable.                                             |

At this checkpoint the production frontend contains only a Slice 0 system-readiness surface. Its lint, strict typecheck, seven unit tests, optimized Next.js 16.2.10 build, and HTTP/CSS browser delivery pass. None of the prototype's business screens is production-ready. The access/audit/system schema and three test-only migrations exist, but development is unmigrated and authentication/business APIs do not.

## 7. Prototype data and behavior that must not cross the boundary

| Prototype gap                                                                                     | Production decision                                                                                           |
| ------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| `DB.stock` disagrees with seven unit records                                                      | Stock is derived from serialized units, batches, and the append-only movement ledger.                         |
| Sales rows disagree with Dashboard/Finance KPIs; margin labels use different denominators         | Reports aggregate canonical posted records and expose metric definitions. Reconciliation tests block release. |
| String joins by customer/supplier/technician name and display-string dates                        | Use foreign keys/IDs and real timestamps; format only at the UI boundary.                                     |
| Whole-rupee JavaScript numbers                                                                    | Persist integer minor units and use verified shared money helpers.                                            |
| Fake IMEIs, PTA references, inspection evidence, landed costs, price histories, and resale values | Remove. Missing evidence/data produces a blocked or honest empty state.                                       |
| In-memory sequences and `Date.now()` IDs                                                          | Use server-issued UUIDs and transactional `number_sequences` for human document numbers.                      |
| `localStorage` digital seed and page-local mutations                                              | Delete. Production state is PostgreSQL-backed and transactionally updated through APIs.                       |
| Global audit rows presented as entity history                                                     | Query append-only `AuditEvent` by entity/scope; keep InventoryMovement as a separate ledger.                  |
| Hardcoded nav/notification/task badges                                                            | Derive counts from permission-filtered APIs.                                                                  |
| Generic links to `finance.html`, `inventory.html`, `unit.html`                                    | Every row carries a real ID and resolves to its source entity route.                                          |
| Toast-only save/export/print/backup/notification controls                                         | Build the action with verified evidence or remove/disable it with honest copy.                                |

## 8. Production UX that should be preserved

- Preserve the prototype's compact, counter-speed POS flow and keyboard-first interaction.
- Preserve ranked, impact-linked owner attention rather than a generic admin dashboard.
- Preserve plain-language impact summaries instead of “Are you sure?” confirmations.
- Preserve status text/icons in addition to color, responsive navigation, dark theme, accessible empty/error/loading states, tabs for entity depth, and timelines for provenance.
- Preserve cost/profit redaction for roles without permission.
- Preserve the explicit separation of digital-service principal, customer fee, provider charge, commission, and net service earnings.
- Preserve visible variance and exceptions; never hide mismatches to make a screen look healthy.

These are experience requirements, not permission to copy the prototype's static implementation.
